Speaker label prefix uses `text.secondary` (#98989D / #6E6E73) color and semibold weight. Per D-05, no color coding per speaker -- all speaker labels use the same secondary text color. This keeps visual hierarchy clean: speaker label (secondary, semibold) followed by segment text (primary, regular).

## Component Inventory

### 1. Speaker Label Prefix (inline modification to TranscriptViewer)

**Location:** Inside `TranscriptViewer._original_edit` (QTextEdit), prepended to each segment line.

**Format per D-05:** `"Speaker 1: Hello everyone"` -- inline text prefix, no badges, no color coding.

**Visual spec:**
- Speaker label: 14px, weight 600, `text.secondary` color (#98989D / #6E6E73)
- Colon + space separator after label
- Segment text: 14px, weight 400, `text.primary` color (existing)
- Each segment on its own line, separated by 8px vertical spacing (sm token)
- Segments without speaker data: display text only, no prefix (per D-07)

**Rendering approach:** QTextEdit with HTML formatting. Each segment rendered as:
```html
<p style="margin-bottom: 8px;">
  <span style="color: #98989D; font-weight: 600;">Speaker 1:</span> Hello everyone
</p>
```

**Click-to-rename (D-06):** Speaker labels are not clickable inside QTextEdit. Instead, a rename mechanism is provided via right-click context menu on the QTextEdit:
- Right-click anywhere in the transcript text
- Context menu includes "Rename Speaker..." option (enabled only when cursor is on a line with a speaker label)
- Opens QInputDialog with current speaker name pre-filled
- On confirm: updates all segments with that speaker label in memory and saves to transcript.json
- Alternatively: a dedicated speaker list panel (see Component 3 below) provides rename functionality

### 2. "Identify Speakers" Button (new, in TranscriptViewer)

**Location:** Export button bar at bottom of TranscriptViewer, left-aligned (export buttons remain right-aligned).

**Layout:** `["Identify Speakers" button] [stretch] [Export SRT] [Export VTT] [Export to Obsidian]`

**Visual spec:**
- QPushButton, fixed width 160px
- Same styling as existing export buttons (control bg, primary text, border default, borderRadius sm)
- Icon: none (text-only, consistent with export buttons)

**States:**
- Default (no diarization data, audio file exists): "Identify Speakers" enabled
- Default (diarization data exists, audio file exists): "Re-identify Speakers" label, enabled
- Disabled (no audio file): "Identify Speakers" grayed, tooltip explains
- Running: "Identifying speakers..." label, disabled, non-interactive
- Complete: reverts to "Re-identify Speakers" label

**Visibility per D-02:** Visible on all transcripts. Disabled with tooltip when audio file is missing.

### 3. Speaker List Panel (new, in TranscriptViewer)

**Location:** Below the title/meta labels, above the tab widget. Collapsible -- only visible when diarization data is present.

**Visual spec:**
- Horizontal flow layout (QHBoxLayout)
- Each speaker shown as: clickable QLabel with speaker name, 14px, `text.secondary`, underline on hover
- Speakers separated by 16px (md token) horizontal spacing
- Prefix label: "Speakers:" in 11px caption, `text.tertiary`
- Maximum display: 8 speakers. Beyond 8, show "+N more" suffix.

**Rename interaction (D-06):**
- Click on any speaker name label
- QInputDialog appears: title "Rename Speaker", prompt "Enter name for {current_name}:", default value is current name
- On accept: rename propagates to all segments with that speaker label, transcript.json is saved
- Speaker list panel updates immediately

**Behavior:**
- Hidden when transcript has no diarization data (per D-07)
- Shows automatically after diarization completes
- Speaker names reflect the latest saved state (user renames or raw SPEAKER_XX labels)

### 4. Status Bar Diarization Progress (D-03)

**Location:** Existing QStatusBar at bottom of MainWindow.

**Visual spec:**
- Uses existing `QLabel[state="processing"]` styling: `status.processing` color (#FF9F0A / #FF9500)
- No additional progress bar -- text-only status message in status bar
- Indeterminate progress conveyed by text content alone (pyannote does not provide granular progress)

**States:**
- Downloading model: "Downloading speaker identification model..."
- Running: "Identifying speakers..."
- Complete: "Speakers identified" (status.success color, 3 seconds then clears)
- Error: "Speaker identification failed: {reason}" (status.error color, persists until user action)

### 5. HuggingFace Token Input (in Settings Dialog)

**Location:** New section in Settings Dialog, under "AI Providers" or as a new "Speaker Identification" section.

**Visual spec:**
- Section heading (14px, semibold): "Speaker Identification"
- HuggingFace token row:
  - Label (14px): "HuggingFace Token"
  - QLineEdit (password mode, echoMode=Password), 280px wide
  - "Save Token" QPushButton (caption size, 80px wide)
  - Status: "Token saved" (success color) or "Required for speaker identification" (text.tertiary)
- Help text below (11px, text.tertiary): "Required for downloading the speaker identification model. Get a token at huggingface.co/settings/tokens"
- Link button: "Get Token" -- opens https://huggingface.co/settings/tokens in browser

**Validation:**
- Token must start with "hf_" (basic format check)
- On save: stored via `keychain.store_api_key("huggingface", token)` per D-11
- Invalid format: "Token must start with 'hf_'" (error color)

### 6. Model Download Progress (D-10)

**Approach:** Reuse existing ModelDownloadThread pattern from onboarding.py.

**Trigger:** First time user clicks "Identify Speakers" and model is not cached.

**Visual spec:**
- Status bar shows: "Downloading speaker identification model... {percent}%"
- Uses `QLabel[state="processing"]` styling
- If HuggingFace token is missing: QMessageBox prompt asking user to add token in Settings
- If model terms not accepted: QMessageBox with instructions to visit the model page and accept terms

## Error and Degradation States

| Scenario | Visual Response | Copy |
|----------|----------------|------|
| No HuggingFace token | "Identify Speakers" button enabled, click triggers QMessageBox | "HuggingFace token required. Add it in Settings > Speaker Identification." |
| HF token invalid / model terms not accepted | Status bar error (persists) | "Model access denied. Accept terms at huggingface.co/pyannote/speaker-diarization-community-1" |
| Audio file missing | "Identify Speakers" disabled, tooltip explains | Tooltip: "Audio file not found -- cannot identify speakers" |
| Diarization fails mid-run | Status bar error, button reverts to enabled | "Speaker identification failed: {error}" |
| Model download fails | Status bar error, button reverts to enabled | "Failed to download model: {error}. Check your internet connection." |
| Re-diarization on transcript with renamed speakers | Warning-free: new labels assigned, user can rename again | Status bar: "Speakers identified" (same as initial) |
| v1.0 transcript loaded | No speaker panel visible, "Identify Speakers" available if audio exists | No special messaging (per D-07) |

## QSS Additions

New QSS rules needed in `ThemeEngine.generate_qss()`:

1. **Speaker name labels** -- clickable labels in speaker list panel:
```
QLabel#speaker_name {
    color: {text.secondary};
    font-size: 14px;
    font-weight: 600;
}
QLabel#speaker_name:hover {
    text-decoration: underline;
}
```

2. **Speaker list prefix** -- "Speakers:" label:
```
QLabel#speaker_list_prefix {
    color: {text.tertiary};
    font-size: 11px;
}
```

Note: QLabel hover pseudo-state is not natively supported in QSS. Speaker name hover effect should be implemented via `enterEvent`/`leaveEvent` with manual font update, or via `setCursor(Qt.CursorShape.PointingHandCursor)` plus underline toggle.
